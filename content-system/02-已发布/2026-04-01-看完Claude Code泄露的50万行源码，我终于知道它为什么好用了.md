前两天我刚在想，Agent 工程里到底什么东西会过时，什么东西不会结果 Claude Code 的源码被泄露出来了这件事一下子把我前面比较抽象的判断，变得非常具体：模型能力当然重要，但真正决定一个 Agent 产品好不好用的，越来越是模型外面那套系统。这套东西，现在越来越适合用一个最近很火的词来概括：Harness Engineering很多人这两天在讨论 Claude Code讨论点主要集中在 Anthropic 怎么又犯低级错误、source map 怎么会被一起打包发出去、1900 个源文件到底暴露了多少细节这次到底算不算意外开源但只把这件事看成一次工程事故或者看成程序员喜闻乐见的大厂翻车，其实有点看小了因为这次泄露真正有价值的地方，是它第一次比较完整地把一个成熟 AI Agent 产品背后的后厨掀开了（相信很多厂商就会借此机会推出自己的Agnet CLI 遥遥领先）以前很多人都在说 Claude Code 好用但好用这个词一直有点模糊你只能大概感觉到它更稳、更像真的在工作、更少乱来、更懂你的上下文很难把这种感觉拆开来讲清楚这次不一样这次外界第一次比较具体地看见了，一套成熟的 AI 编程产品到底是怎么围绕模型搭系统的它更像是把 Agent 时代真正值钱的工程能力，直接摊开给大家看了01.为什么很多人学 Agent，总有一种"学了也会过时"的感觉这两年只要认真碰过 Agent，你大概都会有一种熟悉的焦虑框架、API、写法年年在换去年还在讲的方法，今年可能已经没人提了LangChain 改过好几轮，AutoGen 在变，LlamaIndex 也在重构很多人花时间学了一堆具体接口回头发现半年之后最熟的那套东西已经不重要了所以一个很自然的判断就是：Agent 工程是不是本来就不稳定？这个判断不能说错因为如果你学的主要是框架层、接口层、补丁层的东西，那它确实容易过时以前模型窗口不够大，你要自己做分块、检索、拼接后来窗口大了，很多场景下这些补丁直接被吞掉了以前模型不会稳定调用工具，你要写很多格式约束和解析逻辑后来模型原生支持 tool use，这些代码一下就贬值了以前模型推理弱，你要手把手扶着它走后来模型能力增强，这些扶手又被收回去如果你把 Agent 工程理解成围着模型短板打补丁的活那，它当然会不断变化但 Agent 工程里并不只有这种会被模型吞掉的东西真正不会过时的，在另一层。。。。。02.模型外面的系统，才是真正不会过时的部分我越来越倾向于把 Agent 工程分成两层一层是补偿性工程哪里不够补哪里模型窗口不够就补检索，工具调用不稳就补约束，容易跑偏就补引导这层会随着模型升级不断被吃掉另一层是系统性工程它解决的不是模型还不够强的问题，而是一个智能体只要进入真实环境工作就一定会遇到的问题比如它到底看到了什么上下文，下一步怎么决定做什么，权限边界在哪里，做错了怎么收住，怎么记住真正该记住的东西，怎么在长对话里不丢关键条件，怎么和其他 Agent 协作但不互相污染这些问题不会因为模型更强就自动消失相反，模型越强，你越敢让它做更复杂的事，这些系统问题就越重要所以 Agent 工程真正值钱的部分，开始越来越清楚了：怎么围绕模型搭出一套能让它稳定工作的环境这就是 Harness Engineering 的意思模型负责智能，Harness 负责让这种智能变得可靠上下文怎么组织、权限怎么控制、记忆怎么提取、信息怎么压缩、反馈怎么回流、协作怎么分工这些东西，才越来越像决定产品体验上限的部分03.Claude Code 这次真正暴露出来的，是一整套系统思路Claude Code 这次最顶的地方，不仅仅是告诉大家"它内部也写 TypeScript"真正稀缺的，是它把一整套设计思路暴露出来了上下文不是一句 prompt，是一整套环境拼装很多人以为，自己给 AI 发一句话，它看到的就是这句话但 Claude Code 真正收到的是一个被拼装过的环境：身份定义、安全规则、工具说明、当前目录、系统信息、记忆文件、Git 状态、用户自己的配置文件、MCP server 的描述（这也是为什么你的Token在ClaudeCode里面这么不经烧）全部在发送前组装好，模型看到的是整个宏观的世界这意味着一个产品的表现不只由模型决定，还由你给模型看到了什么世界决定以前很多人把注意力放在"怎么写 prompt"现在越来越重要的问题已经变成了：你到底怎么构造上下文自动化的关键不是胆子大，是边界清楚Claude Code 暴露出来的一大块设计是它背后那套权限分类和安全审查机制很多人理解 AI 自动化，容易把重点放在"让它能做更多事"但！一个成熟系统真正关心的，往往是让它在什么条件下不能乱做事你越想给 Agent 更大权限，越得先把边界设计清楚让人敢开自动模式的前提，是它越界的时候能被拦住记忆系统最值钱的，不是记得多，是记得准Claude Code 的记忆思路里，一个很关键的判断是：记偏好，不记代码这句话看起来很小，但背后很成熟代码会变、路径会变、函数位置会变，今天记住的事实明天可能就成误导但人的偏好、做事习惯、表达风格、项目背景，这些东西更稳定，值得长期沉淀好记忆不是"什么都留"，是知道什么不该留而且从泄露的源码来看，Anthropic 在记忆这件事上走得比大家想象的远源码里有一个叫 KAIROS 的 feature flag，出现了 150 多次，对应一个后台自主运行的 daemon 模式里面有一个叫 autoDream 的机制（AI也会做梦吗？）简单来说就是在用户空闲时，agent 会自动做记忆整合：合并分散的观察、去除逻辑矛盾、把模糊的判断转化成确定性结论做完之后，用户再回来的时候，上下文是干净的而且这个整合任务跑在一个独立的子 agent 里，不会污染主 agent 的思维链这是 agent 在你不用它的时候，自己在消化之前的工作经验上下文压缩不是"总结一下"，是结构化保真对话一长，就开始压缩但真正的风险从来不是压缩本身，是压缩时把关键条件压丢了尤其是用户中途纠正过你的那几句话很多时候后面反复犯错，不是模型不会，是那句关键纠正没有被好好保留下来成熟的压缩，需要明确知道什么必须保留、什么可以丢、什么优先级更高、什么是下一轮决策一定要继续带着的多 Agent 协作越来越像组织设计Claude Code 暴露出来的多 Agent 结构里，有 leader，有 teammate，有异步通信，有独立工作区，有权限上浮，有任务拆分和结果合并多 Agent 真正难的，不是开几个脑子，是怎么分工、怎么协作、怎么避免互相拖垮这已经非常像组织管理问题了（所以用好Agent的前提是学好项目管理。。。）另外还有几个值得注意的小细节源码里有一个反蒸馏机制叫 ANTI_DISTILLATION_CC，开启之后，Claude Code 会在 API 请求里注入假的工具定义如果有人录制 API 流量去训练竞品模型，这些假工具会污染训练数据（这招是真的阴）还有他们在命令行编程工具里藏了一套电子宠物系统叫 Buddy，有物种稀有度、闪光变体、随机属性，第一次孵化时 Claude 会给它写一段"灵魂描述"（今天已经正式上线了，CC里输入/buddy会自动孵化，我的是小鸭，听说是最普通的一只）一个严肃的编程 CLI 工具里藏着这种东西这也说明这个团队在产品上花的心思比表面看到的多04.思维方式在变Claude Code 这次事件真正值得开发者认真看的是它把一种新的工程思维变得非常非常具体了以前写程序遇到 bug，第一反应是重跑一遍、看日志、查报错、改代码但在 Agent 场景里，很多问题并不是模型随机没跑对，而是你没有给够上下文、没有把边界说清楚、没有给它反馈回路、给了太多噪音信息、没有把工具描述好、没有让它处在一个足够稳定的工作环境里所以当你以后用 Claude Code、Codex 或者别的 Agent 去构建自己的系统时，如果中间出了问题，不要第一反应就是让它再试一次很多时候该问的是：是不是我还没有把它做对这件事所需要的环境准备完整？更清楚的上下文、更明确的任务边界、更干净的工具集合、更及时的反馈机制、更合理的权限设计、更少噪音的工作空间Agent 很多时候不是不会，是你还没把那个能让它稳定做对的世界搭给它如果你也在用 Agent 去搭自己的东西，那接下来最值得练的一种能力，可能就是先问自己一句：我到底给它准备了一个什么样的工作环境？
