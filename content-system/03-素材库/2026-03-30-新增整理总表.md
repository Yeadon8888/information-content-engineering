# 素材库

## 2026-03-30，人类与 AI 协作的三次跃迁

### 标题建议
人类与 AI 协作，正在从 Prompt 走向 Harness

### 核心观点
人类和 AI 的协作方式，已经从“怎么说”升级到“给什么信息”，再升级到“设计什么运行环境”。真正拉开差距的，不再只是模型能力，而是为 Agent 设计世界的能力。

### 结构拆解

#### 1. 2023，Prompt Engineering
核心问题是怎么措辞。
把模型当黑箱，用提示词技巧逼近能力边界。
代表方法包括 Chain-of-Thought、Few-shot、温度参数调节。
这些做法有效，但脆弱且不稳定，对措辞和样例顺序高度敏感。

#### 2. 2025，Context Engineering
核心问题变成给 AI 看什么。
决定输出质量的，不只是提问方式，而是上下文里到底装了什么。
关键问题是 Context Rot，也就是上下文腐蚀。
对话一长，早期信息失效、被覆盖、互相干扰，模型表现持续下降。
应对方式包括按需加载信息，压缩历史，上子 Agent 分担上下文压力。

#### 3. 2026，Harness Engineering
核心问题变成怎么给 AI 造一个可工作的世界。
人类不再直接写代码，而是设计环境，定义意图，搭建反馈循环，让 Agent 稳定完成任务。
关键不是指导模型“应该怎么做”，而是通过环境和工具链保证它“必须这样做”。

### 关键例子
- AGENTS.md 控制在约 100 行，作为目录入口，而不是一次性灌满上下文
- 用 linter、结构测试、依赖校验替代规范性提醒
- 持续运行的 Agent 定期扫描偏差，更新评分，自动修复技术债
- Agent 可接入界面、日志、指标、浏览器 DevTools 等真实运行环境
- 形成提交、审查、反馈、修复、升级、合并的自动闭环

### 关键补充句
当 Agent 犯错时，你的第一反应不应该是“再试一次”，而是想一想：环境里缺了什么能力。

### 结论金句
- 模型能力已经不是主要瓶颈，工程能力才是
- 同一个模型，放进不同 Harness，结果可能相差数倍
- 不是更聪明的模型，而是更好的设计
- 人类最终的角色，不是亲自执行，而是设计让 AI 工作的世界
- 当 Agent 犯错时，你的第一反应不应该是“再试一次”，而是想一想：环境里缺了什么能力

## 2026-03-30，Agent 时代的个人知识操作系统

### 标题建议
不要再把知识库当收藏夹了，你需要的是一套会工作的系统

### 核心观点
Agent 时代，知识管理正在从“存资料”升级为“运营一个可持续调用的认知系统”。真正完整的系统，不只有知识库，还包括信息入口、整理机制、调用链路和反馈闭环。dontbesilent 负责接住碎片输入，Obsidian 这一套负责把输入沉淀为可读、可写、可链接、可调用的知识资产，最终支撑内容创作与持续复用。

### 系统拆解

#### 1. 输入层，不要让信息在入口处蒸发
很多内容创作者的问题，不是没有信息，而是信息来得太碎，走得太快。
链接、灵感、想法、截图、任务、半句话判断，往往在白天出现，晚上就消失。
这一层的关键不是“记得记笔记”，而是让 Agent 在信息刚出现时就接住它。

dontbesilent 这一类能力的价值就在这里。
它不是等你整理完再来处理，而是在碎片出现时及时承接，避免有价值的输入在入口处流失。

#### 2. 整理层，知识库不是收藏夹，而是 Agent 的工作台
传统知识管理强调收藏和归档，AI 时代更重要的是调用。
知识管理的本质，不是记住更多，而是让信息在需要的时候自动出现。

Obsidian 这一套的价值，不只是记笔记，而是给 Agent 一个真正可工作的台面。
- 本地 Markdown，Agent 读写成本最低
- 数据在本地，主权和隐私可控
- 双向链接和标签，能把笔记变成知识网络
- 文件系统天然兼容，方便 Agent 直接处理

重点不是 Obsidian 这个名字，而是这种“本地 Markdown 知识库”形态，天然适合作为 Agent 的长期工作环境。

#### 3. 约束层，别让 Agent 乱动你的知识资产
当 Agent 进入知识系统后，关键不只是“能不能操作”，而是“能不能按正确方式操作”。
例如移动、重命名、归档文件时，不能直接粗暴用系统命令处理，否则双链、引用和结构关系会被破坏。

所以知识系统必须给 Agent 配上规则、CLI、YAML 规范、目录结构和 SOP。
这背后的逻辑和 Harness Engineering 是一致的。
不是靠提醒 Agent 记住规则，而是把规则编码进工作环境里。

#### 4. 调用层，让知识真正参与创作
系统真正有价值，不在于你存了多少，而在于创作时能不能调出来。
当白天的碎片被接住，夜里的知识被整理好，AI 才能基于你的知识库做选题、提纲、初稿、复盘和风格延展。

这时候知识库不再是静态仓库，而是创作燃料池。
过去写过的文章、拆解过的案例、积累过的方法论，都会变成可调用上下文。

#### 5. 反馈层，内容产出反过来继续增厚系统
最完整的闭环不是“收集到写作”为止，而是：
碎片输入，结构化整理，内容创作，发布复盘，再沉淀回知识库。

内容不是终点，而是新一轮知识增长的起点。
时间越长，系统越厚，Agent 越懂你，输出也越稳定。

### 两块素材的互补关系
- dontbesilent 解决的是“信息入口层”，让 Agent 主动接住碎片，而不是沉默等待
- Obsidian 工作流解决的是“知识中台层”，把碎片变成可管理、可链接、可调用的资产
- 两者结合后，形成“输入、整理、调用、反馈”四层闭环

### 最值得留下的判断
- 知识管理不再是存储问题，而是调用问题
- Agent 不该只在你提问时出现，它应该在信息产生时就介入
- 本地 Markdown 知识库的价值，在于它能成为 Agent 的工作台
- 真正完整的系统，必须同时有输入、整理、调用、反馈四层
- 未来更有价值的不是会记笔记的人，而是会给 AI 搭知识工作流的人

### 结论金句
- 不要再把知识库当收藏夹了，你需要的是一套会工作的系统
- 真正厉害的 Agent，不是会回答问题，而是会接住信息、整理信息、再把信息变成产出
- Obsidian 不是重点，重点是你有没有给 AI 一个能工作的环境
- 未来最值钱的能力，不是写更多内容，而是设计让内容持续长出来的系统

## 2026-03-30，Stitch 上手的真正分水岭，不是 prompt 更长，而是先把视觉风格切对

### 标题建议
UI 设计师必看，保姆级教程带你 3 步上手 Stitch，5 分钟用 AI 做出高级感 UI 界面

### 核心观点
很多人第一次用 Stitch，最容易走偏的地方，不是不会写 prompt，而是太想一次把需求讲完整。模块、功能点、卖点、页面区块全往里塞，结果信息越来越多，页面却越来越像模板。真正容易把效果拉开的，通常不是“把 prompt 写得更长”，而是先把页面的视觉风格切对。

### 结构拆解

#### 1. 第一轮，先把基础方向跑对
第一轮不要追求惊艳，而是先把品牌、人群、结构和基本气质交代清楚。
重点是得到一个“顺”的基础 hero，信息正确，结构完整，排版不乱。
这一步的目标不是出最终版，而是先确认方向没有跑歪。

#### 2. 第二轮，别补模块，先把它推成品牌首屏
真正的分水岭在这里。
不要继续往下补功能区，也不要继续堆信息，而是直接改页面语言。
把首屏从普通产品页，推进成更有品牌氛围的视觉场景。

关键变化包括：
- 从普通首屏改成全屏背景视觉
- 背景从插画或普通渐变，换成更真实、更有情绪的动态影像感
- 导航和按钮加入轻微玻璃质感
- headline 更像高端杂志封面，而不是功能文案

这一轮改的不是细节，而是整页到底有没有品牌记忆点。

#### 3. 第三轮，把氛围锁成一套风格系统
到了第三轮，再补的就不该是零碎要求，而是把前面的感觉固定成统一审美。

这一轮关注的是：
- 配色体系是否统一
- 标题字体是否有明确气质
- 背景视频是不是唯一视觉深度来源
- 玻璃感轻到什么程度才不廉价
- 留白和构图是否真正极简

这一步的本质，是让页面从“有氛围”变成“有一套风格系统”。

#### 4. 背景视频不一定非得在 Stitch 里做
如果是特别吃情绪的 landing hero，背景视频完全可以先在别的生图、生视频工具里做好，再回到 Stitch 里做结构和排版。

Stitch 最擅长的，不是包办全部视觉生产，而是把：
- 页面结构
- 标题层级
- 按钮材质
- 留白节奏
- 品牌气质
这些元素捏成一个统一页面。

### 最值得留下的判断
- Stitch 的分水岭，不是 prompt 写得多完整，而是你有没有先把视觉方向切对
- 做 landing hero 时，先练风格，不要先练信息堆叠
- 从普通产品页到品牌首屏，差的往往不是功能模块，而是画面语言
- 真正的高级感，不来自元素更多，而来自风格更统一
- AI 设计工具最适合先练 hero section，因为这里最容易看出审美差距

### 可复用方法
可以把这套流程总结成一个 3 轮法：

1. 第一轮，先定品牌、人群、结构、基础气质
2. 第二轮，先推画面语言，不补功能模块
3. 第三轮，锁定配色、字体、材质、留白，形成风格系统
