# 口播脚本参考

文章是展开式表达，口播是推进式表达。写口播时，不要只是把文章句子切短，而是要重新组织节奏、停顿和画面。

## 先选输出形态

### 1. 纯口播稿

适合：

- 知识口播
- 观点表达
- 单人出镜
- 没有复杂演示的内容

推荐结构：

`开头钩子 -> 判断/痛点 -> 解释/案例 -> 一句话回收 -> CTA`

输出格式：

```markdown
## 标题/一句话卖点

## 口播稿
[开头]

[中段1]

[中段2]

[收尾]
```

要求：

- 每句尽量短，读出来不费气
- 多用口语判断句，少用书面解释句
- 一屏只保留一个重点

### 2. 口播提纲

适合作者自己二次发挥，不需要逐字稿。

输出格式：

```markdown
## 视频定位
- 平台：
- 目标观众：
- 核心承诺：

## 节奏提纲
1. 开头抓点：
2. 中段推进：
3. 结果/案例：
4. 收尾 CTA：
```

适合：

- 主播本人表达能力强
- 需要保留临场发挥
- 想先校准方向再写逐字稿

### 3. 分镜脚本

适合：

- 工具演示
- App 功能实测
- 需要 A-roll + B-roll 混剪
- 需要剪辑、字幕、音效提示

推荐格式：

```markdown
## 视频定位
- 平台：
- 时长：
- 风格：
- 核心承诺：

## 分镜脚本

### 镜头A-1｜主播开场
画面：
口播：
剪辑/字幕/音效：

### 镜头B-1｜痛点/演示
画面：
口播：
剪辑/字幕/音效：
```

字段说明：

- `画面`：A-roll、B-roll、录屏、截图、特写、快闪等
- `口播`：主播说的话或画外音
- `剪辑/字幕/音效`：剪辑节奏、字幕重点、SFX、转场

## 口播写法四条硬规则

### 1. 先抓，再讲

前 3 秒优先做以下动作之一：

- 直接下判断
- 抛反常识
- 亮结果
- 说痛点
- 给脑洞

不要先铺背景。

### 2. 一段只推进一个点

文章可以一段里容纳背景、解释、判断。口播不行。每一段最好只做一件事：

- 让人停下来
- 解释一个关键点
- 展示一个结果
- 收回一个判断

### 3. 能展示就别复述

如果画面上能看到功能、结果、对比、按钮、生成物，口播就不要把同样的话重复一遍。口播负责解释“为什么值得看”，画面负责证明“它真的发生了”。

### 4. 留出剪辑锚点

每 10-20 秒最好出现一次变化：

- 情绪升高
- 新画面
- 新结论
- 大字字幕
- 结果揭晓

否则口播会像平铺直叙地念稿。

## 三种常用口播结构

### 痛点 -> 演示 -> 回收

适合工具实测、产品更新、教程类视频。

核心：

1. 先说“以前为什么麻烦”
2. 立刻用画面证明“现在怎么变简单”
3. 最后说“这对谁最有价值”

### 判断 -> 解释 -> 案例 -> 升维

适合观点表达、趋势判断、认知内容。

核心：

1. 先下判断
2. 再解释为什么
3. 给案例
4. 回到更大的意义

### 脑洞 -> 实测 -> Wow Moment -> CTA

适合有戏剧感的 AI 玩法、新奇功能、爆点内容。

核心：

1. 先抛一个让人想看结果的设定
2. 快速进入实操
3. 用结果爆点拉高情绪
4. 用 CTA 把互动收住

## 文章改口播的压缩规则

把文章改成口播时，不按段落改，按层改：

1. 先删掉背景铺垫和重复解释
2. 提炼 1 个核心判断 + 2-3 个支撑点
3. 把长句拆成主播说得出口的短句
4. 给每个支撑点补一个画面或例子
5. 最后加开头钩子和结尾 CTA

常见替换：

- `这意味着` -> `说白了`
- `从本质上来看` -> 直接给结论
- `值得注意的是` -> 删掉
- `首先/其次/最后` -> 换成自然推进

## 分镜写作要点

如果用户给了参考样本，就优先学样本节奏和字段，不要自创格式。

没有样本时，默认遵循：

- 镜头编号清楚
- A-roll 负责人格与判断
- B-roll 负责证明与节奏
- 字幕只放最值得记住的话
- 音效和转场只标必要节点，不要堆满
